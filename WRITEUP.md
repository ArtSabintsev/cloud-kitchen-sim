# Cloud Kitchen Inventory Simulation - Written Response

This follows the assignment's suggested structure. The supporting files are
`PROJECT_SPEC.md`, `AI_USAGE_LOG.md`, `REFLECTION.md`, and the generated
`REPORT.md`.

## 1. Project Setup

The project is organized into these files:

- `seed_data.py`: the starting recipes, inventory, orders, restock records, and
  delivery status records.
- `main.py`: the simulation logic and report generation.
- `test_main.py`: 38 unit tests for the main functions.
- `PROJECT_SPEC.md`: the project specification and business rules.
- `AI_USAGE_LOG.md`: a record of how I used Claude and Codex during the work.
- `REFLECTION.md`: my reflection on AI-assisted coding.
- `REPORT.md`: the Markdown business report generated by the program.

To run the program:

```bash
python3 main.py
```

To run the tests:

```bash
python3 -m unittest -v
```

There are no third-party dependencies. I used Python 3.14, and the assignment
data file was named `seed_data.py` so it could be imported directly by
`main.py`.

## 2. Data Loading

The program has loader and printer functions for all five required seed data
structures: recipes, inventory, orders, restock, and status. I checked the
actual seed data format instead of changing it into a different schema. The
inventory data has one row per ingredient, with quantity in grams and an
`expiry_date` string.

The original restock and status tables are still loaded for the data-loading
tests, but the simulation recalculates live restock and status results when
orders are processed.

## 3. Recipe Lookup

Recipe lookup is handled by `find_recipe_by_name`. It uses exact item-name
matching, so `"Chicken Burger"` matches the `"Chicken Burger"` recipe. If an
item does not have a matching recipe, the program does not crash. It records the
problem and marks the order as not delivered.

Ingredient quantities are calculated with `calculate_ingredient_requirements`.
For example, if an order asks for 2 pizzas, each ingredient amount from the
recipe is doubled. `combine_requirements` then combines repeated ingredients
across the whole order.

## 4. Inventory Availability

`check_inventory_availability` compares the ingredients required by an order
against the current inventory. It checks:

- whether the ingredient exists,
- whether enough grams are available,
- whether the ingredient is expired.

Expired stock is treated as unusable. This matters because the assignment asks
for expiry status to be part of the availability check. The function returns a
clear reason for each ingredient, such as `Available`, `Insufficient quantity`,
`Expired`, or `Not in inventory`.

## 5. Order Fulfillment

`process_orders` uses all-or-nothing fulfillment. If every required ingredient
is available and usable, the order is marked delivered and the inventory is
deducted. If anything is missing, expired, or insufficient, the order is marked
not delivered and no inventory is deducted.

The failure reason is saved in the order result and in the status table. Failed
orders also add their blocking ingredients to the restock recommendations, so a
manager can see what stopped the order.

## 6. Cumulative Processing

Inventory deduction is cumulative. The first delivered order reduces the
available inventory, and later orders are checked against what remains. This is
tested with cases where two orders use the same ingredient and the second order
can fail because the first order used the remaining stock.

In the seed run, 4 of 5 orders are delivered. Order 3 is not delivered because it
requires expired Fettuccine Pasta and expired Chocolate.

## 7. Restock and Expiry Logic

`calculate_restock_needs` applies the assignment's restock rules:

- out of stock means quantity is 0,
- running low means quantity is less than or equal to 1,000 g,
- expiring soon means within 5 days of the simulation date,
- expired means the expiry date is before the simulation date.

The par level is 10,000 g. Low stock is topped up to par. Expired or
expiring-soon stock gets a full 10,000 g replacement recommendation because it
should not be relied on for future orders.

One important fix was preserving multiple restock reasons. For example, an item
can be both expired and out of stock. The code keeps all reasons in a `reasons`
list instead of overwriting one reason with another.

The simulation uses a fixed scenario date of `2026-05-10`. I used a fixed date
because the seed data is clearly designed around the May 2026 expiry dates. This
also makes the test results repeatable.

## 8. Business Summary

Requirement 7 is handled by `build_summary`, `print_summary`, and
`write_markdown_report`. The summary includes:

- number of orders delivered,
- number of orders not delivered,
- final inventory levels,
- restock recommendations,
- expired and expiring-soon ingredients,
- orders that could not be fulfilled and the reason why.

The console output is meant to be readable for a kitchen manager, and
`REPORT.md` gives the same information in a cleaner Markdown report.

I also completed two optional enhancements:

- Option C: dynamic menu item disabling.
- Option D: improved Markdown reporting.

For Option C, menu items are disabled when an ingredient is missing, out of
stock, expired, or not available in enough quantity for one serving.

## 9. Refactoring Notes

Two improvements I made during refactoring were:

1. I moved important business values into constants, such as `PAR_LEVEL_GRAMS`,
   `LOW_STOCK_THRESHOLD_GRAMS`, `EXPIRING_SOON_DAYS`, and `SIMULATION_DATE`.
   This makes the rules easier to find and change.
2. I changed the restock logic from a single-reason approach to a
   multiple-reason approach. This fixed the requirement that an ingredient can
   qualify for more than one restock reason.

I also added Markdown table escaping because an AI review found that a failure
reason containing a pipe character could break the report table.

## 10. AI Usage Summary

I used Claude 4.8 and Codex 5.5 xFast as coding assistants. Claude helped with planning,
drafting code, and writing tests. Codex was used as a second reviewer to look
for missed requirements and edge cases, as well as the code that implemented fixes to some of the bugs that were in the initial project, including one that Claude missed after Claude's initial passthrough.

I did not just accept every AI suggestion. I used tests and the assignment spec
to decide what to keep, change, or reject. For example, the AI review helped
catch the Markdown table issue and the menu-disabling edge case where stock was
above zero but still below one serving.

The full AI process is documented in `AI_USAGE_LOG.md`.
